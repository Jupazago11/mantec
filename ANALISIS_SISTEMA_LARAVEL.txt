1. RESUMEN GENERAL DEL SISTEMA
==================================================

ManTec es una plataforma de gestión de mantenimiento técnico preventivo diseñada
para empresas industriales con múltiples clientes (arquitectura multi-tenant parcial).
Permite registrar inspecciones de activos industriales organizados por cliente,
agrupación, área y elemento. El sistema combina una interfaz web (Blade + Alpine.js)
con una API REST para una aplicación móvil Android (inspectores).

PROPÓSITO PRINCIPAL:
  - Registrar reportes de inspección preventiva por parte de inspectores
  - Administrar activos industriales (elementos, componentes, diagnósticos, condiciones)
  - Generar indicadores y semáforos de estado de ejecución de mantenimiento
  - Módulo adicional: mediciones de espesores (espesor de materiales)
  - Módulo adicional: eventos de cambio de banda (belt events)

STACK TECNOLÓGICO:
  - Backend : Laravel 13.0, PHP 8.3+
  - Base de datos : PostgreSQL
  - Almacenamiento de archivos : Cloudflare R2 (compatible S3 vía league/flysystem-aws-s3-v3)
  - Autenticación web : Laravel Session Auth (formulario de login)
  - Autenticación API : Laravel Sanctum 4.3 (tokens para app Android)
  - Frontend : Vite 8.0 + Tailwind CSS 4.2.2
  - Reactividad UI : Alpine.js 3.x (CDN, no empaquetado)
  - Íconos : Lucide Icons (CDN)
  - Build : Vite con plugin Laravel

DEPENDENCIAS CLAVE (composer.json):
  - laravel/framework: ^13.0
  - laravel/sanctum: ^4.3
  - league/flysystem-aws-s3-v3: ^3.0 (para Cloudflare R2)
  - Dev: phpunit/phpunit ^12.5.12

4. CONTROLADORES — ANÁLISIS DETALLADO
==================================================

--- AuthController.php ---
Responsabilidad: Login/logout del sistema web.

Métodos:
  showLogin() — Retorna vista auth.login. Si ya está autenticado, redirige por rol.
  login()     — Valida credenciales (username + password). Verifica campo 'status'
                 activo en BD. Redirige según rol:
                   admin_cliente, admin, admin_global, superadmin → admin.dashboard
                   inspector     → inspector.reports.create
                   observador*   → admin.dashboard
  logout()    — Invalida sesión, regenera token CSRF, redirige a /login.

NOTA: No usa email para login, usa campo 'username'. Verifica estado activo del usuario.

--- DashboardController.php ---
Responsabilidad: Vista principal del panel administrativo.

index() — Construye colección groupModules según scoping del usuario autenticado.
           Para superadmin/admin_global: todos los clientes.
           Para admin: clientes asignados vía pivot.
           Para admin_cliente: sus clientes propios.
           Retorna vista admin.dashboard.admin con datos agrupados por cliente→grupo→activo.

--- AdminManagedUserController.php (~1073 líneas) ---
Responsabilidad: CRUD completo de usuarios con sistema de permisos granular.

Métodos clave:
  index()   — Lista usuarios filtrados por scope del autenticado. Pagina resultados.
  create()  — Formulario con roles, clientes, grupos, áreas disponibles.
  store()   — Crea usuario con hash de password. Sincroniza permisos en tablas pivot.
               Envuelto en DB::transaction().
  show()    — Vista detalle con permisos actuales.
  edit()    — Formulario de edición con permisos precargados.
  update()  — Actualiza datos y permisos. Sincroniza relaciones pivot (clientes, grupos).
               Envuelto en DB::transaction().
  destroy() — SIEMPRE retorna 403 Forbidden. Eliminación de usuarios deshabilitada.

Métodos auxiliares:
  syncAreaPermissions() — Intencionalmente no-op (sin-operación). Comentario interno
                          indica migración a modelo basado en grupos.

COMPORTAMIENTOS NOTABLES:
  - La eliminación de usuarios está permanentemente bloqueada (siempre 403)
  - syncAreaPermissions() es dead code funcional (no hace nada pero existe)
  - Uso de DB::transaction() para atomicidad en store/update

--- InspectorReportController.php (~1115 líneas) ---
Responsabilidad: Flujo en cascada para registro de reportes de inspección.

Métodos principales:
  index()  — Lista reportes del inspector con filtros de fecha/semana.
  create() — Formulario con selección en cascada: cliente→grupo→área→elemento.
  store()  — Lógica de validación y almacenamiento más compleja del sistema:
    1. Valida cadena de propiedad: cliente→grupo→área→elemento→componente→diagnóstico→condición
    2. Lógica especial 'belt_change': marca cambio de banda, resetea diagnóstico de desgaste
    3. Condición 'detenido': copia recommendation del último reporte no-detenido del elemento
    4. Fusión de duplicados en 24h: si existe reporte del mismo elemento+componente en 24h,
       actualiza el existente en lugar de crear uno nuevo
    5. Upload de archivos a Cloudflare R2 con ruta generada por ReportFilePathBuilder

AJAX endpoints (todos POST):
  groupsByClient()      — Grupos asignados al inspector para un cliente
  areasByGroup()        — Áreas dentro de un grupo
  elementsByArea()      — Elementos activos en un área
  componentsByElement() — Componentes del elemento con diagnósticos disponibles
  diagnosticsByComponent() — Diagnósticos según componente seleccionado
  conditionsByDiagnostic()  — Condiciones válidas para el diagnóstico
  weeklyStatus()        — Estado semanal de diagnósticos de un elemento
  lastRecommendation()  — Última recomendación registrada para condición+elemento

--- AdminPreventiveReportController.php (~2183 líneas) — CONTROLADOR MÁS GRANDE ---
Responsabilidad: Vista y edición de reportes preventivos para administradores.

Métodos principales:
  show()       — Vista por cliente + tipo de activo. Filtra por año, semana, grupo.
                 Aplica scoping por rol (admin ve solo sus clientes, etc.)
  general()    — Vista general de todos los tipos de activo de un cliente.
  showByGroup() — Vista agrupada por grupo con rango de fechas personalizable.

Endpoints de edición inline:
  updateOrdenAviso()    — Solo admin_cliente puede editar campos orden/aviso.
                          Retorna JSON para actualización AJAX.
  updateExecutionDate() — Actualiza fecha de ejecución del detalle.
  adminEdit()           — Edición completa: condición, diagnóstico, fecha, observación,
                          recomendación, estado de ejecución. Solo roles admin.

Métodos privados clave:
  resolveAdminClienteResponsables() — JOIN complejo para encontrar usuarios responsables
                                      admin_cliente por cliente+área.
  applyFilters()        — Aplica filtros de año, semana, grupo, área, elemento, condición,
                          estado de ejecución a query de ReportDetail.
  applyGroupFilters()   — Similar pero orientado a vista por grupo.

--- AuthApiController.php ---
Responsabilidad: Login de la app móvil Android.

login() — Valida username + password vía API.
           Crea token Sanctum con nombre 'android-app'.
           Retorna JSON con: token, user_id, name, role, clients[], allowedElementTypes[].
           Verifica estado activo del usuario antes de emitir token.

--- InspectorOfflineCatalogController.php (~557 líneas) ---
Responsabilidad: Proveer catálogo completo offline para la app Android.

index() — Valida que el inspector tenga exactamente 1 cliente y 1 grupo asignado.
           (Si tiene 0 o 2+, retorna error 422)
           Retorna JSON con catálogo completo:
             - client: datos del cliente
             - group: datos del grupo
             - element_types: tipos de activo habilitados
             - areas: áreas del grupo
             - conditions: condiciones activas del sistema
             - elements: activos con sus relaciones
             - components: componentes disponibles
             - diagnostics: diagnósticos disponibles
             - element_component_relations: relaciones elemento↔componente
             - component_diagnostic_relations: relaciones componente↔diagnóstico
             - component_condition_relations: relaciones componente↔condición
             - measurement_element_types: tipos para módulo mediciones

resolveElementIdsFromGroupPivot() — Método defensivo que prueba 8 nombres posibles
                                    de tabla pivot para encontrar elementos del grupo.
                                    Indica falta de convención estable en pivots.

5. MODELOS Y RELACIONES
==================================================

--- User ---
Traits: HasApiTokens (Sanctum), Notifiable
Relaciones:
  belongsTo: Role
  belongsToMany: Client (pivot: client_user)
  belongsToMany: Group (pivot: group_user o similar)
  hasMany: ReportDetail
  hasMany: MeasurementThicknessDraft (o similar)

Métodos de permisos:
  isPowerAdmin()                    — true si rol es superadmin o admin_global
  canManageSystemModule(key)        — puede gestionar CRUD de un módulo
  canViewSystemModule(key)          — puede ver un módulo
  canCreateInSystemModule(key)      — puede crear registros en un módulo
  hasEnabledModuleForClientAndElementType(clientId, elementTypeId, moduleKey)
  hasCreationEnabledForClientAndElementType(clientId, elementTypeId, moduleKey)

NOTA: Autorización 100% manual con abort_unless(). No usa Gates ni Policies de Laravel.

--- Role ---
Campos: key, name, status
7 roles definidos en RolesSeeder:
  superadmin       — Acceso total, sin restricciones
  admin_global     — Similar a superadmin, acceso multi-cliente
  admin            — Administrador operacional, clientes asignados
  admin_cliente    — Administrador del cliente, acceso limitado
  inspector        — Solo registro de reportes, acceso a app móvil
  observador       — Solo lectura, panel web
  observador_cliente — Solo lectura, limitado a sus clientes

--- Client ---
Relaciones: hasMany Element, hasMany Group, belongsToMany User
Campos notables:
  - auto_sync (boolean): indica si el cliente tiene sincronización automática habilitada
  - slug: usado para construcción de rutas R2
hasDependencies() — Verifica si el cliente tiene datos dependientes antes de eliminar.
                    Retorna bool para prevenir eliminación de clientes con datos.

--- Element ---
Relaciones:
  belongsTo: Area, Group (FK directa group_id), ElementType, Client
  hasMany: ReportDetail
  hasMany: MeasurementThicknessDraft
  hasMany: BeltEventDraft (o similar)

PROBLEMA DETECTADO: Relaciones duplicadas en el modelo Element:
  thicknessDraft()             → apunta a MeasurementThicknessDraft
  measurementThicknessDraft()  → apunta al MISMO modelo MeasurementThicknessDraft
  (Duplicación similar existe para reports)
  Esto genera confusión pero no error en tiempo de ejecución.

--- ReportDetail ---
Campos: report_id, user_id, element_id, component_id, diagnostic_id, year, week,
        condition_id, observation, recommendation, orden, aviso, is_belt_change,
        execution_status_id, execution_date, status

Hook booted():
  Al guardar, si no tiene execution_status_id:
    1. Carga condición (de relación cargada o por condition_id)
    2. Si condición code='ok' → execution_status_id = OK status, execution_date = null
    3. Si otra condición → execution_status_id = PENDING status
    4. Si no hay condición → execution_status_id = PENDING status

NOTA: report_id aparece en $fillable pero observación del código indica que
      en la práctica siempre es null. ReportDetail funciona como registro independiente,
      no como detalle de un "Report" padre.

Relaciones:
  belongsTo: User, Element, Component, Diagnostic, Condition, ExecutionStatus
  hasMany: ReportDetailFile

--- ExecutionStatus ---
Campos: id, name, code, status (boolean activo/inactivo)
Seeder hardcodea: id=1 code='pending' name='PENDIENTE', id=2 code='done' name='REALIZADO'
Resolución: ExecutionStatusResolver usa coincidencia de strings (no IDs hardcoded)

--- Condition ---
Campos: code, name, status
Condición especial: code='ok' → elemento en buen estado (sin ejecución pendiente)
Condición especial: code similar a 'detenido' → copia recomendación anterior

--- Component, Diagnostic ---
Entidades de catálogo técnico.
Relacionados mediante tablas pivot: component_diagnostic, component_condition

--- Group, Area ---
Jerarquía: Client → Group → Area → Element
Group tiene FK directa en Element (además de pasar por Area)

--- ClientElementTypeModule ---
Tabla de configuración de módulos por cliente y tipo de activo.
Campos: client_id, element_type_id, module_id, status, module_enabled
Controla visibilidad del módulo "mediciones" por cliente.

--- SystemModule ---
Catálogo de módulos del sistema (actualmente: 'mediciones')
Campos: key, name, status

7. ROLES, PERMISOS Y ACCESOS
==================================================

SISTEMA DE AUTORIZACIÓN: Manual con abort_unless()
No se usan: Laravel Gates, Laravel Policies, Spatie Permission

7 ROLES DEL SISTEMA:

superadmin
  - Acceso total a todas las funcionalidades
  - Ve todos los clientes, grupos, áreas, activos
  - Puede gestionar todos los módulos
  - Sin restricciones de scoping

admin_global
  - Equivalente a superadmin en acceso
  - En sidebar: misma sección "Global" que superadmin
  - isPowerAdmin() = true

admin
  - Acceso a clientes asignados mediante pivot
  - Gestión operacional (áreas, grupos, activos)
  - NO ve sección Global (Clientes, Usuarios globales)
  - Sí tiene gestión de usuarios de sus clientes
  - Acceso a mediciones si módulo habilitado para sus clientes

admin_cliente
  - Admin restringido al cliente asignado
  - Puede editar campos orden/aviso en reportes preventivos
  - NO puede hacer CRUD de configuración técnica
  - Acceso a mediciones si módulo habilitado

inspector
  - SOLO puede registrar reportes (web + app Android)
  - Sidebar muestra únicamente "Registrar reporte"
  - Login redirige a inspector.reports.create
  - API: recibe token Sanctum para app Android
  - Debe tener exactamente 1 cliente y 1 grupo para catálogo offline

observador
  - Solo lectura
  - Ve Dashboard e Indicadores
  - Sin acceso a configuración ni registro

observador_cliente
  - Como observador pero limitado a sus clientes
  - Ve Dashboard e Indicadores

SCOPING EN CONTROLADORES:
  Cada controlador verifica el rol mediante condicionales explícitas:
  - if($user->isPowerAdmin()) → query sin restricciones
  - elseif($role === 'admin') → filtra por clientes del usuario
  - elseif($role === 'admin_cliente') → filtra por sus clientes propios
  - etc.

  Pattern típico:
    abort_unless(auth()->check(), 401);
    abort_unless($user->canViewSystemModule('mediciones'), 403);

13. FLUJOS PRINCIPALES DEL SISTEMA
==================================================

--- FLUJO 1: REGISTRO DE REPORTE (Inspector Web) ---

1. Inspector hace login con username
2. Redirigido a /inspector/reports/create
3. Formulario carga: selección de cliente (pre-asignado)
4. AJAX: obtiene grupos del cliente → /ajax/groups-by-client
5. AJAX: obtiene áreas del grupo → /ajax/areas-by-group
6. AJAX: obtiene elementos del área → /ajax/elements-by-area
7. AJAX: obtiene componentes del elemento → /ajax/components-by-element
8. AJAX: obtiene diagnósticos del componente → /ajax/diagnostics-by-component
9. AJAX: obtiene condiciones del diagnóstico → /ajax/conditions-by-diagnostic
10. Inspector completa: observación, recomendación, fotos
11. POST /inspector/reports → InspectorReportController@store
12. Sistema verifica: cadena de propiedad cliente→...→condición
13. Sistema aplica lógica especial:
    - Si is_belt_change: marca y resetea diagnóstico
    - Si detenido: copia última recomendación
    - Si duplicado en 24h: actualiza en lugar de crear
14. ReportDetail.booted() auto-asigna execution_status
15. Archivos subidos a Cloudflare R2
16. Redirección a lista de reportes

--- FLUJO 2: REGISTRO VÍA APP ANDROID ---

1. Inspector usa app Android
2. POST /api/login → obtiene token Sanctum
3. GET /api/inspector/offline-catalog → descarga catálogo completo
4. App trabaja offline (SQLite local)
5. Inspector registra reportes offline
6. Al reconectar: POST /api/inspector/reports (cada reporte)
7. POST /api/inspector/reports/{id}/sync-files (archivos)

--- FLUJO 3: VISTA DE SEMÁFORO (Admin) ---

1. Admin navega a Indicadores
2. Selecciona cliente + tipo de activo
3. GET /admin/preventive-reports/show?client=X&elementType=Y&year=Z&week=W
4. AdminPreventiveReportController@show consulta ReportDetail
5. Agrupa por elemento → semana → componente → estado
6. Determina color semáforo según ExecutionStatus y SemaphoreTemplate
7. Renderiza tabla con semáforo visual por elemento/semana

--- FLUJO 4: EDICIÓN INLINE DE REPORTE (Admin) ---

1. Admin ve tabla de semáforo con íconos de edición
2. Click en ícono → modal/formulario inline
3. PATCH /admin/preventive-reports/details/{id}/execution-date
   o PUT  /admin/preventive-reports/details/{id}/admin-edit
4. Actualización sin reload de página (AJAX)
5. Semáforo actualiza color si cambió el estado de ejecución

--- FLUJO 5: HABILITACIÓN DE MÓDULO MEDICIONES ---

1. superadmin accede a /admin/client-element-type-modules
2. Selecciona cliente + tipo de activo + módulo
3. Habilita module_enabled = true
4. Sidebar del cliente objetivo ahora muestra "Mediciones"
5. Inspector/admin_cliente pueden registrar mediciones

--- FLUJO 6: AUTO-ASIGNACIÓN DE ESTADO DE EJECUCIÓN ---

Triggered en: ReportDetail.booted() → evento 'saving'
1. Si execution_status_id ya existe → no hace nada
2. Carga condición relacionada
3. Si condición code='ok':
   - execution_status_id = ID del estado "OK"
   - execution_date = null (sin fecha de ejecución pendiente)
4. Si otra condición:
   - execution_status_id = ID del estado "PENDIENTE"
5. Si no hay condición:
   - execution_status_id = ID del estado "PENDIENTE"

15. POSIBLES MEJORAS, FALLOS E INCONSISTENCIAS
==================================================

--- INCONSISTENCIA 1 (CRÍTICA): RUTAS DEBUG SIN AUTH EN PRODUCCIÓN ---
Las rutas /test-r2 y /php-upload-check no tienen middleware de autenticación.
Exponen información sensible: configuración de R2, parámetros PHP de upload.
Riesgo: Cualquier persona puede acceder a esta información en producción.
Acción recomendada: Remover o proteger con middleware auth + rol superadmin.

--- INCONSISTENCIA 2 (ALTA): ASSETS CDN SIN VERSIÓN FIJA ---
Alpine.js usa @3.x.x (fija major, no minor/patch).
Lucide usa @latest (sin ninguna fijación de versión).
Si el CDN actualiza la librería con breaking changes, el sistema puede romperse
sin aviso y sin un despliegue propio.
Acción recomendada: Instalar vía npm y empaquetar con Vite, o fijar versión exacta en CDN.

--- INCONSISTENCIA 3 (ALTA): RELACIONES DUPLICADAS EN MODELO Element ---
Element tiene dos relaciones que apuntan al mismo modelo:
  - thicknessDraft() → MeasurementThicknessDraft
  - measurementThicknessDraft() → MeasurementThicknessDraft
Y duplicación similar para "reports".
Genera ambigüedad en el código y posible confusión para quien extienda el modelo.
Acción recomendada: Eliminar el alias duplicado, mantener solo la relación con nombre descriptivo.

--- INCONSISTENCIA 4 (ALTA): CAMPO report_id SIEMPRE NULL ---
ReportDetail tiene report_id en $fillable y en la tabla, pero no se asigna en ningún flujo.
Sugiere diseño incompleto: había planeado un modelo "Report" agregador que nunca se implementó.
Esto puede crear confusión sobre la arquitectura del sistema.
Acción recomendada: Documentar si report_id es deprecated o planificar su implementación/eliminación.

--- INCONSISTENCIA 5 (ALTA): PIVOT DE ELEMENTOS POR GRUPO INESTABLE ---
InspectorOfflineCatalogController::resolveElementIdsFromGroupPivot() prueba 8 nombres
de tabla pivot porque no hay convención estable.
Esto indica que el schema de la BD puede haber cambiado varias veces sin limpiar las
referencias, o que hay múltiples convenciones coexistiendo.
Acción recomendada: Estandarizar y documentar el nombre correcto de la tabla pivot. Remover los intentos fallidos.

--- INCONSISTENCIA 6 (MEDIA): AUTORIZACIÓN 100% MANUAL ---
No se usan Gates, Policies ni paquetes como Spatie Permission.
Cada controlador reimplementa la lógica de autorización con condicionales manuales.
Riesgo: Cualquier nuevo controlador puede olvidar verificar permisos.
Acción recomendada: Migrar a Policies de Laravel para centralizar y testear la autorización.

--- INCONSISTENCIA 7 (MEDIA): syncAreaPermissions() ES DEAD CODE ---
En AdminManagedUserController, syncAreaPermissions() existe pero no hace nada.
El comentario indica que "se migró a modelo basado en grupos" pero el método vacío
permanece, causando confusión sobre si está pendiente de implementación o es intencional.
Acción recomendada: Eliminar el método o marcar explícitamente con TODO/deprecation comment.

--- INCONSISTENCIA 8 (MEDIA): destroy() DE USUARIO SIEMPRE RETORNA 403 ---
AdminManagedUserController::destroy() retorna 403 sin excepción ni condición.
No es un error, es diseño intencional (usuarios no se eliminan), pero no está documentado
en el código. Quien lo lea asumirá que es un bug.
Acción recomendada: Agregar comentario explicativo o eliminar el endpoint del router completamente.

--- INCONSISTENCIA 9 (MEDIA): IDs HARDCODED EN ExecutionStatusSeeder ---
El seeder fuerza id=1 para PENDIENTE y id=2 para REALIZADO.
Si la BD se reinicia o migra y los IDs cambian, cualquier código que use IDs directos fallará.
ExecutionStatusResolver evita IDs, pero puede haber código que los use directamente.
Acción recomendada: Usar código/key en lugar de ID en cualquier referencia directa a execution_statuses.

--- INCONSISTENCIA 10 (BAJA): SIDEBAR CON CÓDIGO PHP DUPLICADO ---
resources/views/components/admin/sidebar.blade.php tiene $hasMeasurementsEnabledConfig = false;
asignado DOS veces (línea 62 y línea 64).
Esto es inofensivo pero indica copy-paste sin limpieza.
Acción recomendada: Eliminar la línea duplicada.

--- INCONSISTENCIA 11 (BAJA): CONTROLADOR PREVENTIVO DE 2183 LÍNEAS ---
AdminPreventiveReportController.php tiene ~2183 líneas, lo que excede ampliamente
el límite recomendado de responsabilidad única.
Combina: filtros de query, resolución de responsables, lógica de visualización,
         validación de accesos, formateo de datos.
Acción recomendada: Extraer servicios/queries a clases separadas (sin urgencia si funciona bien).

--- SemaphoreBuilder.php ---
  Inyecta: LegacySemaphoreDefinition, SemaphoreColumnOrderer.
  Responsabilidad: construir las celdas del semáforo (colores y estados).

  legacyColumns()
    Retorna las 5 columnas hardcoded del semáforo legacy:
    change_belt, belt_status, safety_condition, discharge, cleaner

  serializeTemplateColumns(SemaphoreTemplate $template)
    Serializa columnas de una plantilla dinámica para el frontend.
    Aplica ordenamiento vía SemaphoreColumnOrderer.

  buildLegacyRowCells(Collection $details, ?SemaphoreBeltChange $override)
    Construye celdas para el semáforo LEGACY (hardcoded):
    - change_belt: cambio de banda (con override manual via SemaphoreBeltChange)
    - belt_status: estado de banda (basado en condición del componente 'Banda')
    - safety_condition: seguridad (componentes: Guardas, Cubiertas, Plataforma)
    - discharge: descarga (componente: 'Tolva de alimentación')
    - cleaner: limpiador (componentes: Primario, Secundario, Transversal, en V)

  buildTemplateRowCells(Collection $details, ?SemaphoreBeltChange $override, SemaphoreTemplate $template)
    Construye celdas para plantillas dinámicas.
    Resuelve columnas con dependencias entre sí (source_column_key).
    Detecta ciclos de dependencia y retorna estado 'neutral' si hay ciclo.
    Para column_type='condition_aggregate': agrega condiciones por reglas (component+diagnostic)
    Para column_type='belt_change_manual': usa SemaphoreBeltChange override o ReportDetail

--- SemaphoreColumnOrderer.php ---
  Responsabilidad: Ordenar columnas de semáforo respetando dependencias source_column_key.
  orderCollection(Collection $columns)
    Mueve columnas de tipo 'belt_change_manual' antes de su columna fuente (source_column_key).
    Garantiza que la columna dependiente esté antes de la columna de la que depende en el render.
  normalizePayload(array $columns)
    Normaliza un payload de columnas (desde formulario/API):
    - Calcula key slug si no se provee
    - Aplica el mismo reordenamiento
    - Asigna position numérico final

--- LegacySemaphoreDefinition.php ---
  Define las columnas y componentes del semáforo ANTERIOR (pre-plantillas dinámicas).
  Hardcodea nombres de componentes para cada columna:
    dischargeComponentNames():   ['Tolva de alimentación']
    cleanerComponentNames():     ['Limpiador primario', 'Limpiador secundario',
                                  'Limpiador transversal', 'Limpiador en V']
    safetyComponentNames():      ['Guardas de seguridad', 'Cubiertas', 'Plataforma y estructura']
    beltStatusComponentNames():  ['Banda']
  RIESGO: Si los nombres de componentes en la BD difieren (mayúsculas, tildes, espacios),
          el semáforo legacy no funcionará correctamente.
  COEXISTENCIA: Ambos sistemas (legacy y dinámico por plantillas) coexisten en el código.

--- AppServiceProvider.php ---
  boot(): URL::forceScheme('https') — fuerza HTTPS en todas las URLs generadas.
  register(): vacío (ningún binding custom registrado aquí).
  NOTA: Los servicios Semaphore se inyectan vía inyección de constructor
        (Laravel auto-resolve sin binding explícito en el provider).

22. MIDDLEWARE PERSONALIZADOS
==================================================

ESTADO: No existe ningún middleware personalizado en app/Http/Middleware/.

El sistema usa exclusivamente los middleware integrados de Laravel:
  - 'auth': Verifica sesión activa (redirige a /login si no está autenticado)
  - 'auth:sanctum': Verifica token Sanctum para rutas API
  - Los middleware base de Laravel Framework (CSRF, Session, etc.)

OBSERVACIÓN: Toda la lógica de autorización (roles, scoping) está implementada
directamente en los controladores con abort_unless() en lugar de middleware dedicado.

Esta elección significa:
  + Más simple de leer en cada controlador (la autorización está visible)
  - Difícil de reutilizar (mismo check repetido en múltiples controladores)
  - Sin un punto central para auditar qué roles tienen acceso a qué
  - No aprovecha el pipeline de middleware de Laravel para autorización

24. INCONSISTENCIAS ADICIONALES IDENTIFICADAS
==================================================

--- INCONSISTENCIA 12: SISTEMA LEGACY Y DINÁMICO COEXISTIENDO ---
El sistema de semáforo tiene DOS implementaciones paralelas:
  1. LEGACY (LegacySemaphoreDefinition): columnas hardcoded basadas en nombres de componentes
  2. DINÁMICO (SemaphoreTemplate + columns + rules): columnas configurables por BD
IndicatorController usa SemaphoreBuilder que soporta ambos (buildLegacyRowCells y buildTemplateRowCells).
No hay separación clara de cuándo se usa uno u otro.
RIESGO: El legacy depende de nombres exactos ('Tolva de alimentación', 'Limpiador primario', etc.)
        Si los datos en BD tienen ortografía diferente, el semáforo legacy no funcionará.

--- INCONSISTENCIA 13: DOS CONTROLADORES DE USUARIO ---
Existen UserController.php (~494 líneas) Y AdminManagedUserController.php (~1073 líneas).
Ambos gestionan usuarios pero con diferente alcance y complejidad.
No está claro si UserController es código legacy o si se usa actualmente.
Si no se usa: dead code. Si se usa: lógica de negocio dividida.

--- INCONSISTENCIA 14: TABLA PIVOT OBSOLETA ---
user_client_element_type_areas (migración 2026-04-07) parece reemplazada por
user_client_group_areas (migración 2026-04-28).
La tabla antigua puede seguir existiendo en la BD sin uso.
resolveElementIdsFromGroupPivot() en InspectorOfflineCatalogController busca en 8 nombres
posibles — probablemente incluye intentos a la tabla obsoleta.

--- INCONSISTENCIA 15: CAMPOS ALTER EL MISMO DÍA ---
En 2026-04-19 y 2026-04-21 se hicieron ALTER el mismo día de la creación de la tabla:
  - create_client_element_type_modules + alter (olvido de creation_enabled)
  - create_band_events + add_real_fields (olvido de ~14 campos)
  - create_band_event_drafts + add_real_fields (idem)
Esto indica diseño incompleto antes de implementar la migración.
En producción: ambas migraciones deben ejecutarse en orden correcto.

--- INCONSISTENCIA 16: VISTAS EN admin/inspections/ SIN RUTAS ---
El explorador encontró vistas en resources/views/admin/inspections/ (create, edit, index, show)
que no corresponden a ninguna ruta registrada en web.php.
Posiblemente: código legacy de un módulo anterior, o rutas eliminadas sin limpiar las vistas.

--- INCONSISTENCIA 17: URL::forceScheme('https') EN AppServiceProvider ---
Activa HTTPS forzado globalmente incluyendo en ambiente de desarrollo.
Si un desarrollador nuevo configura el proyecto en HTTP local, todas las URLs
generadas serán https:// causando errores de mixed content o redirect loops.
Debería condicionar: if ($this->app->environment('production')) { URL::forceScheme('https'); }

--- INCONSISTENCIA 18: SEMAPHORE TEMPLATE COLUMN RULES — ELIMINACIÓN TOTAL EN UPDATE ---
En AdminSemaphoreTemplateController::syncColumns(), al actualizar una columna:
  $column->rules()->delete(); — elimina TODAS las reglas existentes
  Luego las recrea desde el payload.
Este patrón elimina y recrea en lugar de hacer diff/sync.
En tablas pequeñas no importa, pero si hay muchas reglas puede ser ineficiente.
Además, si el payload llega incompleto (error de red), se pierden las reglas anteriores.

ElementSeeder.php define 'K91BT04' tanto en 'MOLINO CRUDO' como en 'MOLINO CEMENTO'.

Esto puede causar:
  - Reportes de inspección duplicados para el mismo activo físico
  - Inconsistencias en el semáforo (dos filas para K91BT04)
  - Confusión para el inspector en el formulario de reporte

==================================================
30. LÓGICA DE NEGOCIO — CONTEXTO DETALLADO
==================================================

Este sistema resuelve un problema muy concreto del mundo industrial:
las plantas de producción (cementeras, mineras, plantas de proceso) tienen
cientos de equipos mecánicos que deben inspeccionarse periódicamente. Sin un
sistema estructurado, los hallazgos se pierden en papel, los técnicos no tienen
trazabilidad de qué revisaron y los administradores no saben qué está pendiente
de mantenimiento. ManTec digitaliza ese proceso completo.

==================================================
30.1 EL MUNDO REAL QUE REPRESENTA EL SISTEMA
==================================================

Una planta industrial como CORONA (cementera) tiene áreas físicas: TRITURACION,
PREHOMO, APILADO DE ADITIVOS, MOLINO DE CARBON, etc. Dentro de cada área hay
activos industriales (en el caso demo: bandas transportadoras como '211BT01',
'311BT01', 'L21BT01', etc. — nomenclatura alfanumérica propia de la planta).

Cada banda transportadora (Element) tiene componentes físicos:
  - La banda misma (Banda)
  - Chutes de descarga
  - Limpiadores (primario, secundario, transversal, tipo arado)
  - Rodillos (de carga, de retorno, laterales)
  - Tambores (de cola, de inflexión, de contra pesa, snub, motriz)
  - Motorreductor
  - Guardillas, Encausadores, Cama de impacto
  - Condición de seguridad, Cubiertas, Material acumulado, Otros

Cada componente puede presentar distintos diagnósticos clínicos:
  - Estado (general)
  - Empalme (solo para la Banda)
  - Alineacion
  - Temperatura
  - Pantalla/Lámina de sacrificio (solo para Chute)
  - Recubrimiento y Rodamientos (para los Tambores)
  - Aseo (para Material acumulado)

Cada diagnóstico en un momento dado tiene una condición:
  - OK (verde, severity=0): todo bien, no requiere intervención
  - BAJA (azul, severity=3, código PF3): falla menor
  - MEDIA (amarillo, severity=2, código PF2): atención próxima
  - ALTA (rojo, severity=1, código PF1): intervención urgente
  - SEG C / SEG D: condiciones de seguridad (crítica y media)
  - ASEO (naranja): requiere limpieza
  - OBSERVACIÓN (violeta): punto de seguimiento

El inspector visita la planta semanalmente y para cada banda, en cada componente,
registra qué diagnóstico aplica y cuál es la condición actual. Eso es un ReportDetail.

==================================================
30.2 LA JERARQUÍA COMPLETA DE DATOS
==================================================

La estructura jerárquica del sistema refleja exactamente la organización física
de una planta industrial:

NIVEL 1 — CLIENTE (Client)
  Empresa cliente que contrató los servicios de ManTec.
  Ejemplo: CORONA. Tiene su propio conjunto de activos, usuarios y configuración.
  Un cliente puede tener múltiples grupos, áreas y tipos de activo.
  El campo 'auto_sync' (boolean) indica si este cliente tiene habilitada la
  sincronización automática con la app Android del inspector.

NIVEL 2 — AGRUPACIÓN (Group)
  Subdivision lógica dentro del cliente. Puede representar una planta, una línea
  de producción, un edificio, o cualquier agrupación funcional que el cliente defina.
  Un grupo contiene múltiples áreas y múltiples elementos (FK directa).
  Tiene su propio 'auto_sync' independiente del cliente.
  Los inspectores se asignan a grupos específicos — esto determina qué activos
  pueden inspeccionar desde la app Android (catálogo offline).

NIVEL 3 — ÁREA (Area)
  Zona física dentro de la planta: TRITURACION, PREHOMO, MOLINO CRUDO, etc.
  Las áreas pertenecen a un cliente y contienen elementos.
  La jerarquía completa para un elemento es: Client → Group → Area → Element.
  IMPORTANTE: Group y Area son independientes entre sí (no hay FK Group→Area).
  Un Element tiene FK tanto a Area (area_id) como a Group (group_id).
  Esto significa que la consistencia "este elemento está en esta área que está
  en este grupo" NO está enforced por la BD — es responsabilidad del operador
  mantener la coherencia al crear elementos.

NIVEL 4 — ACTIVO (Element)
  El equipo físico inspeccionable. En el caso demo: cada banda transportadora
  identificada por su código de planta (211BT01, K91BT04, L21SM01, etc.).
  Campos clave:
    - name: nombre/código del activo
    - code: código interno (puede ser igual a name)
    - warehouse_code: código de almacén (nullable — no todos lo tienen)
    - element_type_id: tipo de activo (define qué componentes/diagnósticos aplican)
    - area_id: área física donde está ubicado
    - group_id: grupo al que pertenece (duplica información si área ya está en grupo)
    - status: si está activo para inspección

NIVEL 5 — TIPO DE ACTIVO (ElementType)
  Define la "plantilla" del activo. Todos los activos del mismo tipo comparten
  el mismo catálogo de componentes, diagnósticos y condiciones.
  Ejemplo: todos los activos de tipo "Banda transportadora" tienen los mismos
  22 componentes y 9 diagnósticos disponibles.
  El campo 'has_semaphore' indica si este tipo de activo participa en el semáforo visual.

NIVEL 6 — COMPONENTE (Component)
  Parte física del activo que se inspecciona. Los componentes son del tipo de activo,
  no del activo individual (aunque se asignan a cada activo via element_components).
  Campos clave:
    - is_required: el inspector DEBE reportar este componente (campo informativo, no enforced)
    - is_default: el componente aparece pre-seleccionado en el formulario
    - code: código técnico del componente (nullable — no todos tienen código propio)

NIVEL 7 — DIAGNÓSTICO (Diagnostic)
  El aspecto técnico que se está evaluando del componente.
  Un componente puede tener múltiples diagnósticos (via component_diagnostics).
  Ejemplo: el componente 'Banda' tiene diagnósticos: Estado, Empalme, Alineacion, Temperatura.
  El inspector selecciona el diagnóstico y luego la condición resultante.

NIVEL 8 — CONDICIÓN (Condition)
  El resultado de la inspección: el estado actual del diagnóstico en ese componente.
  Tiene código, nombre, severidad numérica y color.
  La condición code='ok' es especial: significa que todo está bien, no se requiere acción.
  Otras condiciones generan una tarea de mantenimiento pendiente.

==================================================
30.3 EL CICLO SEMANAL DE INSPECCIÓN
==================================================

El sistema opera en semanas ISO 8601. Cada reporte tiene year + week (no fecha exacta).
Esto significa que la unidad mínima de tiempo para el sistema de semáforos es la semana.

PASO 1 — EL INSPECTOR SE PREPARA
  El inspector hace login (web o app Android).
  En la app Android: descarga el catálogo offline completo antes de salir a campo.
  El catálogo incluye todos los datos necesarios para trabajar sin internet:
  cliente, grupo, áreas, elementos, componentes, diagnósticos, condiciones.
  Restricción: el inspector debe tener exactamente 1 cliente y 1 grupo asignado
  para poder usar el catálogo offline. Si tiene 0 o 2+, la API rechaza la descarga.

PASO 2 — EL INSPECTOR RECORRE LA PLANTA
  Para cada activo (Element) que visita:
    a. Selecciona el componente que está revisando
    b. Selecciona el diagnóstico aplicable a ese componente
    c. Evalúa la condición actual (OK, BAJA, MEDIA, ALTA, etc.)
    d. Escribe observación (descripción del hallazgo)
    e. Escribe recomendación (qué hacer para corregirlo)
    f. Puede adjuntar fotos/videos como evidencia
    g. Puede marcar 'is_belt_change' si está registrando un cambio de banda

  Cada combinación (elemento + componente + diagnóstico + condición + semana)
  genera un ReportDetail independiente. Un inspector puede generar decenas de
  ReportDetail en una sola jornada de inspección.

PASO 3 — LÓGICA ESPECIAL DE CONDICIÓN 'DETENIDO'
  Si la condición seleccionada contiene la palabra 'detenido' (case-insensitive),
  el sistema activa un comportamiento especial:
    - Busca el último ReportDetail del mismo elemento+componente que NO sea 'detenido'
    - Copia automáticamente la recomendación de ese reporte anterior al nuevo
  Propósito: cuando un activo está detenido, la recomendación vigente sigue siendo
  la del último hallazgo real, no "el activo está detenido".

PASO 4 — LÓGICA DE FUSIÓN EN 24 HORAS
  Si el inspector (o la app) genera un ReportDetail para el mismo
  (element_id + component_id) dentro de las últimas 24 horas, el sistema NO
  crea un registro nuevo — actualiza el existente.
  Propósito: evitar duplicados cuando un inspector corrige un registro recién creado,
  o cuando la app sincroniza un reporte que ya fue enviado parcialmente.
  El resultado es que en la tabla report_details no pueden existir dos registros
  del mismo elemento+componente con menos de 24h de diferencia del mismo inspector.

PASO 5 — AUTO-ASIGNACIÓN DE ESTADO DE EJECUCIÓN
  Al guardar cualquier ReportDetail (via modelo booted() hook):
    Si condición code = 'ok':
      → execution_status = Estado "OK" (verde)
      → execution_date = null (no hay fecha de ejecución — no se necesita intervención)
    Si cualquier otra condición:
      → execution_status = Estado "PENDIENTE"
      → execution_date = null (pendiente de ser ejecutado)
    Si no hay condición:
      → execution_status = "PENDIENTE"

  Esto significa que INMEDIATAMENTE al registrar un hallazgo con condición diferente
  a OK, el sistema lo marca como PENDIENTE de mantenimiento, listo para ser visto
  en el semáforo del administrador.

PASO 6 — SINCRONIZACIÓN (APP ANDROID)
  Si el inspector trabajó offline:
    - Al conectarse, la app envía cada ReportDetail via POST /api/inspector/reports
    - Los archivos se sincronizan via POST /api/inspector/reports/{id}/sync-files
    - El servidor aplica la misma lógica de negocio (validación de cadena, fusión 24h,
      auto-asignación de estado)
    - La app recibe el ID del registro creado/actualizado para confirmar sincronización

==================================================
30.4 EL SEMÁFORO — CORAZÓN VISUAL DEL SISTEMA
==================================================

El semáforo es la vista más importante para los administradores. Permite ver
de un vistazo el estado de mantenimiento de todos los activos de un cliente.

ESTRUCTURA DEL SEMÁFORO:
  Filas: cada activo (Element)
  Columnas: cada componente o agrupación de componentes
  Celdas: el estado de ese componente en ese activo para la semana/período seleccionado

CÓMO SE DETERMINA EL COLOR DE CADA CELDA:

  1. El sistema busca todos los ReportDetail del activo para el período indicado
  2. Filtra por el componente de la columna
  3. Toma el diagnóstico con la condición de mayor severidad (severity menor = más crítico:
     severity=1 es ALTA, severity=2 es MEDIA, severity=3 es BAJA)
  4. El color de la celda es el color de esa condición (campo Condition.color)
  5. Si no hay ningún reporte → celda vacía (color definido por empty_state_behavior
     de la columna: 'neutral' = gris, 'ok' = verde)

SEMÁFORO LEGACY vs DINÁMICO:

  LEGACY (existente antes de mayo 2026):
    Columnas hardcodeadas en LegacySemaphoreDefinition:
      - change_belt: cambio de banda (manual, con override via SemaphoreBeltChange)
      - belt_status: estado de banda (componente 'Banda')
      - safety_condition: seguridad (componentes: Guardas de seguridad, Cubiertas,
                          Plataforma y estructura)
      - discharge: descarga (componente: Tolva de alimentación)
      - cleaner: limpiador (componentes: Limpiador primario, secundario, transversal, en V)
    La lógica de cada columna está hardcodeada en SemaphoreBuilder::buildLegacyRowCells().

  DINÁMICO (implementado en mayo 2026 con las migraciones de semaphore_templates):
    Las columnas se configuran por el administrador via CRUD de Plantillas de Semáforo.
    Cada plantilla define columnas de dos tipos:
      - condition_aggregate: agrega condiciones de múltiples pares (componente+diagnóstico)
        y muestra el peor estado encontrado según severity_direction (asc: menor=peor, desc: mayor=peor)
      - belt_change_manual: muestra el estado de cambio de banda (manual via SemaphoreBeltChange
        o automático desde ReportDetail con is_belt_change=true)
    Una plantilla puede ser de scope global (solo client+elementType) o específica
    de un grupo (client+group+elementType).
    Solo puede haber UNA plantilla is_default=true por scope.

  COEXISTENCIA:
    IndicatorController usa SemaphoreBuilder que soporta ambos modos.
    Si el ElementType tiene plantilla activa → usa el semáforo dinámico.
    Si no hay plantilla → usa el semáforo legacy.
    (La lógica exacta de selección depende del IndicatorController, no documentada
    en detalle por ser el controlador más largo del sistema: 1,703 líneas).

EL CAMBIO DE BANDA EN EL SEMÁFORO:
  La columna 'change_belt' (legacy) o tipo 'belt_change_manual' (dinámico) es especial.
  Cuando un inspector registra is_belt_change=true en un ReportDetail, eso indica
  que la banda fue cambiada físicamente en esa semana.
  Adicionalmente, los administradores pueden registrar o corregir manualmente el
  estado de cambio de banda via SemaphoreBeltChange (tabla: semaphore_belt_changes
  con campos: element_id, year, week, changed=boolean).
  El override manual tiene precedencia sobre el ReportDetail.
  Propósito: el cambio de banda es un evento de alto costo que quiere visualizarse
  independientemente del estado general del componente.

INDICADORES vs SEMÁFORO vs REPORTES PREVENTIVOS:
  Existen 3 vistas diferentes orientadas al mismo tipo de datos:

  admin.indicators.* (IndicatorController):
    Vista dinámica principal. Permite seleccionar cliente, grupo, tipo de activo y
    rango de fechas. Carga datos via AJAX JSON (no recarga de página). Muestra el
    semáforo visual con colores. Soporta edición del cambio de banda por el admin.

  admin.preventive-reports.show (AdminPreventiveReportController@show):
    Vista por cliente + tipo de activo + año/semana específica. Más estática.
    Permite edición inline de campos orden/aviso (solo admin_cliente) y
    edición completa (solo roles admin). Muestra evidencias fotográficas.

  admin.preventive-reports.by-group (AdminPreventiveReportController@showByGroup):
    Vista agrupada por grupo con rango de fechas personalizable.
    Diseñada para ver el estado de toda la planta de un cliente en un período.

==================================================
30.5 EL CICLO DE EJECUCIÓN DE MANTENIMIENTO
==================================================

Una vez que el inspector registra un hallazgo, comienza el ciclo de ejecución:

ESTADO 1: PENDIENTE
  Creado automáticamente al registrar cualquier ReportDetail con condición != 'ok'.
  execution_status = PENDIENTE, execution_date = null.
  Visible en el semáforo con el color de la condición (rojo, amarillo, azul, etc.).
  El administrador ve qué activos tienen mantenimiento pendiente.

ESTADO 2: SEGUIMIENTO ADMINISTRATIVO
  Los administradores pueden ver en la tabla de reportes preventivos:
    - La descripción del hallazgo (observation)
    - La recomendación del inspector (recommendation)
    - Los campos orden y aviso (únicamente editables por admin_cliente):
        - orden: número de orden de trabajo (OT) del sistema de mantenimiento externo
        - aviso: número de aviso del sistema externo (ej: SAP)
      Estos campos permiten vincular el hallazgo en ManTec con la orden de trabajo
      en el sistema ERP/CMMS de la empresa cliente.

ESTADO 3: EJECUCIÓN
  Cuando el trabajo de mantenimiento es realizado:
    - El administrador registra la execution_date (fecha de ejecución)
    - El execution_status cambia a EJECUTADO/REALIZADO
    - El semáforo para ese activo+componente+semana cambia a verde (ejecutado)
  Este cambio lo hace AdminPreventiveReportController@updateExecutionDate o
  @adminEdit (según el nivel de edición requerido).

ESTADO 4: VERIFICACIÓN
  El inspector, en la siguiente visita, vuelve a evaluar el mismo componente.
  Si el trabajo fue efectivo → condición = OK → nuevo ReportDetail con execution_status = OK.
  Si el trabajo no fue suficiente → nueva condición de falla → nuevo ReportDetail PENDIENTE.
  El historial completo queda en report_details (todos los registros, semana a semana).

CAMPOS orden Y aviso — INTEGRACIÓN CON ERP EXTERNO:
  Estos dos campos representan el punto de integración con el sistema de gestión
  de mantenimiento del cliente (típicamente SAP PM, Maximo, etc.).
  - 'orden': número de orden de trabajo (Work Order) en el ERP del cliente.
  - 'aviso': número de aviso o notificación en el ERP (SAP usa avisos como precursores
    de órdenes de trabajo).
  Al registrar estos números, el administrador del cliente puede cruzar información
  entre ManTec y su sistema ERP para saber si la orden fue ejecutada y cerrada.
  La edición de estos campos está RESTRINGIDA a admin_cliente (no admin ni superadmin).
  Esto refleja que el admin_cliente es el responsable de gestionar las órdenes de
  trabajo en el sistema del cliente, no el personal de ManTec.

==================================================
30.6 EL MÓDULO DE MEDICIONES DE ESPESOR
==================================================

Paralelo al sistema de reportes de inspección existe el módulo de mediciones,
diseñado para registrar mediciones cuantitativas (no cualitativas como los reportes).

PROPÓSITO:
  Medir el espesor físico de la banda transportadora en múltiples puntos.
  La banda se desgasta con el tiempo — medir el espesor permite predecir cuándo
  necesitará reemplazo antes de que falle en operación.

ESTRUCTURA DE MEDICIÓN:
  Una medición tiene "cubiertas" (capas de la banda). Para cada cubierta (cover_number),
  se miden 9 puntos en una grilla 3x3:
    - top_left,    top_center,    top_right    (cubierta superior, 3 puntos)
    - bottom_left, bottom_center, bottom_right (cubierta inferior, 3 puntos)
    - hardness_left, hardness_center, hardness_right (dureza, 3 puntos)
  Todos los valores son decimales con 2 cifras de precisión (milímetros presumiblemente).

FLUJO DRAFT → REPORT:
  1. El técnico crea o abre el borrador del activo (Draft: 1 por activo, UNIQUE en BD)
  2. Agrega líneas al borrador (una por cubierta numerada)
  3. Cuando la medición está completa, publica el borrador como Report (inmutable)
  4. El Draft queda libre para la próxima medición
  5. Los Reports publicados forman el historial de espesores del activo a lo largo del tiempo

PERMISOS DE ACCESO:
  El módulo requiere dos condiciones simultáneas:
    a. El ROL del usuario tiene canViewSystemModule('mediciones') = true
    b. El CLIENTE del activo tiene module_enabled=true en ClientElementTypeModule
       para el módulo 'mediciones' y el ElementType correspondiente
  Si ambas condiciones se cumplen, el sidebar muestra "Mediciones".
  Si el rol además tiene canCreateInSystemModule('mediciones') = true Y el cliente
  tiene creation_enabled=true, el usuario puede crear nuevas mediciones.
  Si solo puede ver (no crear), puede consultar el historial pero no agregar.

ACCESO DESDE APP ANDROID:
  El módulo de mediciones también está disponible desde la app móvil via
  InspectorMeasurementThicknessController (API).
  El catálogo offline incluye los measurement_element_types habilitados.
  Permite al técnico medir en campo sin conexión y sincronizar después.

30.7 EL MÓDULO DE EVENTOS DE BANDA
==================================================

Los eventos de banda registran intervenciones físicas específicas sobre
la banda transportadora. Son eventos discretos en el tiempo, no inspecciones semanales.

TIPOS DE EVENTO:
  band: Cambio completo de la banda
    Datos: marca, espesor total, espesor cubierta superior/inferior, cantidad de pliegues,
           ancho, largo, cantidad de rollos, tipo de vulcanizado, condiciones de entrega
           (corriente del motor, alineación, acumulación de material, guardas, idlers).

  vulcanization: Vulcanización (empalme o reparación de la banda)
    Datos: tipo de vulcanizado, temperatura, presión, tiempo, tiempo de enfriamiento.

  section_change: Cambio de tramo (reemplazo de una sección de la banda)
    Datos: marca del tramo, espesor, cantidad de pliegues, largo y ancho del tramo.

FLUJO DRAFT → PUBLISH:
  1. Se crea un BandEventDraft del tipo requerido (1 borrador por tipo por activo)
  2. El técnico completa los campos del borrador (puede guardarlo parcialmente)
  3. Puede adjuntar evidencias (fotos/videos) al borrador via BandEventDraftEvidence
  4. Al publicar: BandEventDraft → BandEvent (registro definitivo)
  5. BandEventEvidence se crea a partir de las evidencias del draft
  6. El draft queda disponible para el próximo evento del mismo tipo

  NOTA: El parent_id en BandEventDraft y BandEvent permite modelar la relación entre
  un evento y el BandEvent que lo origina. Por ejemplo, una vulcanización tiene como
  'padre' el cambio de banda al que corresponde.

RELACIÓN CON EL SEMÁFORO:
  Cuando el inspector registra is_belt_change=true en un ReportDetail (en el flujo
  de inspección semanal), eso señala el cambio de banda en el contexto del semáforo.
  Los BandEvent son el registro técnico detallado del mismo evento.
  Ambos sistemas son paralelos — el ReportDetail marca la semana del cambio para
  el semáforo, el BandEvent guarda todos los datos técnicos del reemplazo.

ESTADO DE LA BANDA (BandStateDraft / BandStateReport):
  Paralelo a los eventos, se registra el estado dimensional general de la banda:
  ancho (width), espesor cubierta superior (top_cover), espesor cubierta inferior
  (bottom_cover) y descripción del estado.
  Este es el "estado actual" de la banda, mientras que los eventos son el historial
  de intervenciones. El último BandStateReport del activo se muestra en la vista del
  módulo de mediciones (MeasurementController::levelOne).

30.8 GESTIÓN DE USUARIOS Y SCOPING DE ACCESO
==================================================

El sistema tiene una arquitectura de acceso en capas concéntricas:

CAPA 1 — ROL (global, en toda la plataforma)
  El rol determina qué secciones del sistema puede ver y qué acciones puede realizar.
  Es el filtro más grueso. Se asigna en users.role_id.

CAPA 2 — CLIENTE (scoping por empresa)
  Los usuarios no-superadmin están asignados a clientes específicos via client_user.
  Un admin puede estar en múltiples clientes. Un inspector, en uno solo (prácticamente).
  Solo ven datos de sus clientes asignados.

CAPA 3 — GRUPO (scoping por agrupación)
  Los usuarios pueden estar asignados a grupos específicos dentro de sus clientes
  via group_user. Esto es más granular que el cliente — un inspector puede ver
  solo los activos del Grupo A pero no del Grupo B, aunque ambos sean del mismo cliente.

CAPA 4 — TIPO DE ACTIVO (scoping por ElementType)
  Via user_client_element_type, un usuario (especialmente inspector) puede estar
  habilitado para reportar solo ciertos tipos de activo de un cliente.
  Esto permite que un inspector de bandas no pueda reportar sobre, por ejemplo,
  activos de tipo "Grúa puente" aunque estén en el mismo cliente.

CAPA 5 — ÁREA (scoping granular, legacy)
  Via user_client_group_areas, asignación granular a áreas específicas dentro de grupos.
  Esta tabla reemplazó a user_client_element_type_areas (que está obsoleta).
  Permite decir: "este usuario puede ver los activos del Área TRITURACION
  dentro del Grupo G1 del Cliente CORONA".

COMPORTAMIENTO POR ROL:

  superadmin / admin_global:
    Sin restricciones. Ven todos los clientes, grupos, áreas, activos.
    Pueden crear/editar/eliminar cualquier cosa.
    isPowerAdmin() = true.

  admin:
    Ve solo sus clientes asignados (client_user).
    Dentro de esos clientes, ve toda la jerarquía.
    Puede gestionar usuarios de sus clientes (admin_cliente, inspector, observador).
    Puede crear/editar áreas, grupos, activos de sus clientes.

  admin_cliente:
    Rol del lado del cliente (trabaja en la empresa, no en ManTec).
    Ve sus clientes asignados.
    Solo puede editar campos 'orden' y 'aviso' en los reportes preventivos.
    No tiene acceso a configuración técnica (componentes, diagnósticos, condiciones).
    Puede ver mediciones si el módulo está habilitado.

  inspector:
    Solo puede registrar reportes de inspección (web o app Android).
    No puede ver indicadores, dashboard general, ni configuración.
    Sidebar muestra únicamente "Registrar reporte".
    Scoped a su cliente + grupo + tipos de activo asignados.
    En la app: catálogo offline restringido a 1 cliente + 1 grupo.

  observador / observador_cliente:
    Solo lectura. Puede ver Dashboard e Indicadores.
    No puede registrar reportes ni editar nada.
    Scoped a sus clientes asignados.

ASIGNACIÓN DE INSPECTORES — FLUJO COMPLETO:
  1. superadmin/admin crea usuario con rol 'inspector'
  2. Le asigna un cliente (client_user)
  3. Le asigna el tipo de activo habilitado (user_client_element_type con client_id)
  4. (Opcionalmente) le asigna grupo (group_user) para el catálogo offline
  5. El inspector puede hacer login en web o app Android
  6. En app: descarga catálogo offline del cliente+grupo asignado
  7. Puede inspeccionar todos los activos del tipo asignado en su grupo

30.9 CONFIGURACIÓN TÉCNICA — PLANTILLAS DE CATÁLOGO
==================================================

El catálogo técnico (componentes, diagnósticos, condiciones) es específico
por cliente + tipo de activo. No es global. Esto significa:

  - El cliente CORONA tiene sus 22 componentes específicos (Banda, Chute, etc.)
  - Si hubiera un cliente HOLCIM, tendría sus propios componentes
  - No hay confusión entre catálogos de diferentes clientes

EL FLUJO DE CONFIGURACIÓN DE UN NUEVO CLIENTE:

  1. Crear Cliente (admin.clients.create)
  2. Crear Tipo de Activo para el cliente (admin.managed-element-types)
  3. Crear Componentes del tipo de activo (admin.managed-components)
  4. Crear Diagnósticos del tipo de activo (admin.managed-diagnostics)
  5. Asociar Diagnósticos a Componentes (admin.managed-component-diagnostics)
     → Esto define qué diagnósticos puede tener cada componente
  6. Crear Condiciones del tipo de activo (admin.managed-conditions)
  7. Crear Áreas del cliente (admin.managed-areas)
  8. Crear Agrupaciones (admin.managed-groups)
  9. Crear Activos y asignarlos a área+grupo (admin.managed-elements)
  10. Los activos heredan automáticamente los componentes del tipo de activo
      (vía ElementComponentSeeder o asignación manual)
  11. Crear usuarios y asignarlos al cliente
  12. (Opcional) Crear plantilla de semáforo (admin.semaphore-templates)
  13. (Opcional) Habilitar módulo de mediciones (admin.client-element-type-modules)

PLANTILLAS DE SEMÁFORO — CONFIGURACIÓN AVANZADA:

  La plantilla de semáforo define exactamente qué columnas aparecen en la vista
  de indicadores para un tipo de activo. Para cada columna se configura:

  - label: nombre visible de la columna ('Estado banda', 'Limpiador', etc.)
  - column_type: cómo calcular el estado
      'condition_aggregate': agrupa por reglas (pares componente+diagnóstico)
      'belt_change_manual': estado del cambio de banda
  - severity_direction: cómo ordenar por criticidad
      'asc': severity numérico 1 es el peor (ALTA=1 > MEDIA=2 > BAJA=3)
      'desc': el mayor valor numérico es el peor
  - empty_state_behavior: qué mostrar cuando no hay datos
      'neutral': celda gris/vacía
      'ok': celda verde (implica que ausencia de hallazgo es estado OK)
  - rules: los pares (componente, diagnóstico) que determinan el estado de la columna
      Ejemplo: columna 'Estado banda' tiene regla (Banda, Estado)
               columna 'Limpiadores' tiene reglas (Limpiador primario, Estado),
               (Limpiador secundario, Estado), etc.
  - source_column_key: para columnas 'belt_change_manual', qué columna referencia
      Permite que el cambio de banda se vincule al estado de la banda

  Una plantilla puede ser específica de un grupo o global para el tipo de activo.
  Si existe una plantilla para el grupo específico, se usa esa; si no, la global.

30.11 LA API MÓVIL — FLUJO OFFLINE-FIRST
==================================================

La aplicación móvil Android opera con un modelo offline-first:
el inspector puede trabajar sin conexión a internet y sincronizar después.

AUTENTICACIÓN:
  POST /api/login con username+password.
  Genera token Sanctum con nombre 'android-app'.
  El token se guarda localmente en la app para las siguientes peticiones.
  No hay expiración configurada en el código (Sanctum permite tokens permanentes).

DESCARGA DEL CATÁLOGO:
  GET /api/inspector/offline-catalog (auth:sanctum)
  Valida: inspector activo, exactamente 1 cliente, exactamente 1 grupo.
  Devuelve toda la información necesaria para trabajar offline:
  {
    client: {...},
    group: {...},
    element_types: [...],     ← tipos de activo habilitados para el inspector
    areas: [...],             ← áreas del grupo
    conditions: [...],        ← condiciones activas (OK, ALTA, MEDIA, etc.)
    elements: [...],          ← activos con sus datos
    components: [...],        ← componentes disponibles
    diagnostics: [...],       ← diagnósticos disponibles
    element_component_relations: [...],    ← qué componentes tiene cada activo
    component_diagnostic_relations: [...], ← qué diagnósticos puede tener cada componente
    component_condition_relations: [...],  ← qué condiciones puede tener cada componente
    measurement_element_types: [...]       ← tipos habilitados para mediciones
  }

TRABAJO OFFLINE:
  La app almacena el catálogo localmente (presumiblemente SQLite en el dispositivo).
  El inspector puede registrar múltiples ReportDetail sin conexión.
  Cada registro incluye: element_id, component_id, diagnostic_id, condition_id,
  observation, recommendation, is_belt_change, fotos.

SINCRONIZACIÓN:
  Al recuperar conexión:
  1. POST /api/inspector/reports — envía cada ReportDetail
     El servidor aplica la misma lógica que el flujo web:
     validación de cadena de propiedad, fusión en 24h, auto-asignación de estado.
     Responde con el ID del registro creado/actualizado.
  2. POST /api/inspector/reports/{id}/sync-files — envía archivos de evidencia
     Se suben a Cloudflare R2 con la misma ruta generada por ReportFilePathBuilder.
  3. Si el inspector tiene mediciones pendientes:
     Se sincronizan via InspectorMeasurementThicknessController.

ALMACENAMIENTO DE ARCHIVOS EN CLOUDFLARE R2:
  Todo archivo adjunto (evidencia fotográfica) se sube a Cloudflare R2.
  La ruta en el bucket sigue el patrón de ReportFilePathBuilder::build():
    clientes/{client-slug-id}/agrupaciones/{group-slug-id}/{year}/semana-{week}/{element-slug-id}/{uuid}.{ext}
  Ejemplo:
    clientes/corona-1/agrupaciones/planta-norte-3/2026/semana-15/211bt01-47/a1b2c3d4-e5f6.jpg
  El campo ReportDetailFile.disk guarda el nombre del disco ('r2').
  Para ver el archivo, AdminReportEvidenceController genera una URL temporal (10 min)
  con la misma respuesta Content-Disposition 'inline' para verlo en el navegador.

30.12 RELACIONES ENTRE MÓDULOS — MAPA COMPLETO
==================================================

Los módulos no son totalmente independientes — interactúan entre sí:

ReportDetail ↔ Semáforo:
  Los ReportDetail con year+week alimentan el color del semáforo.
  execution_status en el ReportDetail determina si el hallazgo está pendiente o ejecutado.
  El administrador actualiza execution_date en el ReportDetail → semáforo cambia color.

ReportDetail (is_belt_change) ↔ SemaphoreBeltChange ↔ BandEvent:
  El is_belt_change=true en ReportDetail señala la semana del cambio en el semáforo.
  SemaphoreBeltChange permite al admin corregir/confirmar ese estado manualmente.
  BandEvent es el registro técnico detallado del mismo cambio físico.
  Tres sistemas distintos para el mismo evento físico (cambio de banda).

MeasurementThicknessReport ↔ levelOne view:
  El último MeasurementThicknessReport de cada activo se muestra en la
  vista de selección de activo del módulo de mediciones.
  Permite al técnico ver cuándo fue la última medición de cada banda.

BandStateReport ↔ levelOne view:
  El último BandStateReport también se muestra en la misma vista.
  Permite ver el estado dimensional actual de cada banda sin abrir el detalle.

ClientElementTypeModule ↔ Sidebar ↔ API:
  La configuración del módulo (module_enabled) controla:
    - Si el link "Mediciones" aparece en el sidebar web
    - Si el catálogo offline incluye ese elementType para mediciones
    - Si el inspector puede registrar mediciones via API

SemaphoreTemplate ↔ IndicatorController:
  El IndicatorController consulta si existe una plantilla activa para el
  ElementType del activo. Si existe → semáforo dinámico. Si no → legacy.

30.13 INTEGRIDAD Y VALIDACIONES DE NEGOCIO
==================================================

El sistema implementa múltiples niveles de validación para garantizar
que los datos sean consistentes con la realidad de negocio:

VALIDACIÓN DE CADENA DE PROPIEDAD (InspectorReportController::store):
  Antes de crear un ReportDetail, el sistema verifica:
    1. El cliente pertenece al inspector (client_user)
    2. El grupo pertenece al cliente (groups.client_id)
    3. El área pertenece al cliente (areas.client_id)
    4. El elemento pertenece al área Y al grupo (elements.area_id + elements.group_id)
    5. El componente está habilitado para ese tipo de activo (element_components)
    6. El diagnóstico es válido para ese componente (component_diagnostics)
    7. La condición es válida para ese diagnóstico (component_conditions)
  Si cualquier paso falla → abort(422) con mensaje descriptivo.
  Esto garantiza que ningún inspector pueda reportar sobre activos que no son suyos.

VALIDACIÓN UNICIDAD DE PLANTILLA DE SEMÁFORO:
  AdminSemaphoreTemplateController valida:
    - Nombre único por (client_id, group_id, element_type_id)
    - Solo 1 plantilla is_default=true por ese scope
    - Claves de columna únicas dentro de la plantilla
    - Pares (component_id, diagnostic_id) únicos dentro de una columna
    - El componente y diagnóstico de cada regla pertenecen al mismo cliente+elementType
    - El diagnóstico está asociado al componente (component_diagnostics)
  Estas validaciones garantizan que el semáforo calcule correctamente.

PREVENCIÓN DE ELIMINACIÓN CON DEPENDENCIAS:
  Los modelos con hasDependencies() implementan verificación antes de eliminar:
    Client.hasDependencies(): verifica que no tenga datos relacionados
    Area.hasDependencies():   verifica que no tenga elementos
    Group.hasDependencies():  verifica que no tenga elementos
    Component.hasDependencies(): verifica elements, diagnostics, reportDetails
    Condition.hasDependencies(): verifica reportDetails
  Los controladores correspondientes llaman hasDependencies() antes de proceder.
  Esto previene la eliminación de registros que dejarían huérfanos en otras tablas.

IDEMPOTENCIA EN SEEDERS:
  Los seeders críticos usan updateOrCreate (no insert) para ser re-ejecutables.
  BeltComponentDiagnosticSeeder usa syncWithoutDetaching (no reemplaza relaciones existentes).
  ElementComponentSeeder usa sync (reemplaza completamente — podría borrar relaciones manuales).
  ElementSeeder verifica existencia antes de crear (if exists, continue).

TRANSACCIONES EN OPERACIONES CRÍTICAS:
  AdminManagedUserController::store() y update() usan DB::transaction().
  AdminSemaphoreTemplateController::store() y update() usan DB::transaction().
  Esto garantiza atomicidad: si falla la sincronización de permisos, todo
  el usuario/plantilla se revierte y no queda en estado inconsistente.

30.14 EL PATRÓN DRAFT/REPORT — FILOSOFÍA DE DISEÑO
==================================================

El sistema usa el patrón Draft/Report en todos los módulos que requieren
edición antes de publicar:

  MeasurementThicknessDraft  → MeasurementThicknessReport
  BandStateDraft             → BandStateReport
  BandEventDraft             → BandEvent

La filosofía detrás de este patrón:

DRAFT (Borrador):
  - Mutable: se puede editar las veces que se quiera
  - 1 borrador por entidad clave (UNIQUE constraint en BD)
  - No forma parte del historial oficial
  - Puede quedar incompleto indefinidamente
  - El técnico puede abandonar el trabajo y retomarlo después

REPORT (Reporte publicado):
  - Inmutable: una vez publicado, no se modifica (sin métodos de update en la API)
  - Forma parte del historial oficial permanente
  - Tiene published_at timestamp para auditoría
  - Múltiples reports por entidad (historial completo)

TRANSICIÓN Draft → Report:
  Al publicar:
    1. Se copian los datos del draft al report
    2. El draft queda "vacío" o se reinicia para la próxima medición
    3. Los archivos adjuntos del draft se migran al report (draft_evidences → evidences)

EXCEPCIÓN — ReportDetail (inspecciones semanales):
  Las inspecciones NO usan el patrón Draft/Report. Un ReportDetail es directamente
  el registro final. Esto es porque el inspector evalúa y registra en tiempo real
  durante la visita — no hay un borrador de "voy a inspeccionar".
  La lógica de "fusión en 24h" cumple parcialmente el rol del draft: si el inspector
  necesita corregir su registro, lo hace dentro de las primeras 24h y el sistema
  actualiza el mismo registro en lugar de crear uno nuevo.

30.15 FLUJO ECONÓMICO Y DE NEGOCIO — CÓMO MONETIZA
==================================================

Inferido del diseño del sistema (no documentado explícitamente en el código):

ManTec es una empresa de servicios de mantenimiento industrial que:
  1. Presta el servicio de inspección técnica a sus clientes (CORONA, etc.)
  2. Sus inspectores visitan las plantas de los clientes semanalmente
  3. El sistema ManTec es la herramienta que usan para registrar y entregar los informes
  4. Los admin_cliente de CORONA pueden ver los resultados de las inspecciones
  5. El admin_cliente es el encargado de gestionar las órdenes de trabajo (orden/aviso)
     en el sistema interno de CORONA para corregir los hallazgos
  6. ManTec cobra por el servicio de inspección, no por el software per se

FLUJO DE VALOR:
  ManTec (inspectores) → inspecciona activos de CORONA → registra hallazgos en el sistema
  → CORONA (admin_cliente) ve los hallazgos → crea órdenes de trabajo en su ERP
  → Mantenimiento de CORONA ejecuta los trabajos → admin actualiza execution_date
  → ManTec vuelve a inspeccionar → verifica si el trabajo fue efectivo (nueva condición)
  → Ciclo continuo de inspección → hallazgo → ejecución → verificación

ROLES DESDE LA PERSPECTIVA DE NEGOCIO:
  superadmin/admin_global: Equipo técnico de ManTec (desarrolladores, gerencia)
  admin: Coordinadores de campo de ManTec (coordinan inspectores por cliente)
  admin_cliente: Personal de CORONA encargado de mantenimiento (el "cliente" real)
  inspector: Técnicos de ManTec que van a campo a inspeccionar
  observador: Gerentes o supervisores de CORONA que solo quieren ver indicadores
  observador_cliente: Similar, pero el campo de visión limitado a ciertos clientes

  BT = Banda Transportadora, BP = Banda Plana, SM = Sistema de Medición (o similar)
